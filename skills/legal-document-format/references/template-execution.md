# 模板执行

模板执行是本 Skill 的主路径。它面向 Claude Code、Codex 或兼容 Agent，而不是要求普通用户先把 Word 模板改造成字段工程。

## 核心定义

当用户提供 DOCX 模板、母版文书、机构格式或样本文书时：

- 模板是格式事实来源；
- 用户的自然语言要求是任务指令；
- 用户提供的材料是内容来源；
- 交付前门禁是验收标准。

因此，Agent 的目标不是“找到并替换字段”，而是按模板完成文书生成、批处理、格式迁移或候选文书校验。

## 标准流程

1. 读取任务目标。
   - 单份生成；
   - 批量生成；
   - 将既有文本套入模板；
   - 只做格式检查；
   - 裁决书、律师函、报告、合同附件等特定文书。
2. 读取模板事实。
   - 页眉、页脚、页码字段；
   - 标题层级、字体、字号；
   - 正文段落、行距、缩进；
   - 表格、编号、分节；
   - 落款、签名区、附件区；
   - 特殊样式、字体回退、兼容性设置。
3. 读取内容来源。
   - 用户自然语言；
   - 表格；
   - JSON；
   - Markdown；
   - 已脱敏案情材料；
   - 既有 DOCX 候选稿。
4. 选择执行方式。
   - 模板已有明确 `{{KEY}}` 字段：可用确定性替换脚本；
   - 模板没有字段但结构清晰：由 Agent 生成候选文书并用门禁校验；
   - 模板复杂且位置无法证明：要求用户确认、人工标注或缩小任务范围。
5. 交付前检查。
   - 文本标点和空格；
   - DOCX 结构；
   - 页眉页脚、页码字段和分节；
   - 标题字体字号和标点字体；
   - DOCX -> PDF -> PNG 渲染；
   - 批量任务的逐份 PASS / WARNING / FAIL 汇总。

## 用户自然语言示例

```text
这是律师函模板。
请根据表格里的 20 行数据生成 20 份律师函。
标题、页眉页脚、页码、落款和签名区都按模板。
生成后逐份跑格式检查。
```

```text
这是裁决书格式模板。
请把我提供的事实、争议焦点和裁判理由整理成同版式文书。
不要改变法律判断。
生成后检查字体、字号、行距、页码和中文标点。
```

```text
这是我们公司报告模板。
请把这份 Markdown 改成同版式 DOCX。
如果有无法自动确认的版式，请列出来让我人工确认。
```

## 确定性模式的位置

`{{KEY}}` 字段不是普通用户主路径，而是稳定批量生产时的 deterministic mode。

适用条件：

- 团队已经有固定字段清单；
- 模板长期复用；
- 需要大批量替换；
- 需要证明 OpenXML 结构未漂移；
- 技术同事愿意维护模板字段。

不适用：

- 用户只是临时给一份样本文书；
- 模板没有字段；
- 任务是把一批材料整理成同版式文书；
- 用户不懂也不想维护模板变量。

## 报告要求

模板执行报告必须说明：

- 使用的模板路径；
- 用户任务目标；
- 内容来源；
- 是否使用确定性字段替换；
- 已生成文件；
- 已运行检查；
- 自动确认通过的格式项；
- 需要人工复核的位置；
- 无法承诺的能力边界。

## 不得承诺

- 不得把“按模板生成候选稿”说成“任意无标注模板 100% 自动精确填充”。
- 不得把渲染成功说成法律内容正确。
- 不得在未检查页眉页脚、页码和视觉渲染前说“可正式交付”。
- 不得把真实案件、客户信息或私有模板写入公开仓库。
