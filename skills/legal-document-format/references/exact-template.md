# 精确模板套版

精确模板套版意味着输出文档继承用户提供的 DOCX 母版结构。它不是视觉仿制。

## Base-Replace 原则

当用户提供模板或母版文档时：

1. 以母版 DOCX 作为起始文件。
2. 优先要求模板中放置明确的 `{{KEY}}` 占位符。
3. 只替换必须改变的文本占位符。
4. 除非用户明确授权，否则保留模板自身的结构和元数据。
5. 交付前运行模板一致性检查、结构检查和视觉检查。

V2 默认实现是“复制模板 DOCX 包结构 + 替换文本节点”。这意味着页眉、页脚、页码字段、分节、样式、编号、页边距、段落属性和字体设置都应从模板继承，而不是重新生成。

## V2 工具链

标准流程：

1. 准备带 `{{KEY}}` 占位符的模板 DOCX。
2. 用 `scripts/apply_docx_template.py` 生成候选 DOCX。
3. 用 `scripts/compare_docx_template_parity.py` 对比模板与候选，要求除文本节点内容外的 OpenXML 结构一致。
4. 用 `scripts/render_docx.sh` 渲染候选 DOCX。
5. 用 `scripts/format_gate.py --require-visual --fail-on-warning` 做发布档门禁。

示例：

```bash
./skills/legal-document-format/scripts/apply_docx_template.py \
  模板.docx 输出.docx \
  --replacements-json replacements.json

./skills/legal-document-format/scripts/compare_docx_template_parity.py \
  模板.docx 输出.docx
```

`replacements.json` 示例：

```json
{
  "TITLE": "民事起诉状",
  "CASE_NO": "(2026) 示例案号 001 号"
}
```

## 占位符规则

- 占位符使用 `{{KEY}}` 形式。
- 建议 `KEY` 使用大写字母、数字和下划线，例如 `{{CLAIMANT}}`。
- 默认只保证替换位于同一个 Word 文本节点内的占位符。
- 如果 Word 把一个占位符拆到多个 run 或多个文本节点，未解析占位符门禁会阻断交付。
- 如果需要替换整段、表格行或复杂块，应先把模板设计为稳定占位符块，再执行人工复核。

## 一致性判定

模板一致性门禁忽略文本节点内容差异，但会检查：

- DOCX 包内 part 集合；
- XML 结构、属性和关系；
- 页眉、页脚、页码字段和分节属性；
- 样式、编号、设置、主题和字体表；
- 非文本二进制部件的哈希。

因此，“一模一样”的工程含义是：除被明确替换的文本内容外，文档结构和版式载体不发生漂移。若替换后的文字长度导致分页变化，仍需通过渲染页检查和人工复核确认。

## 默认保留

默认保留：

- 分节和分节属性；
- 页面尺寸和页边距；
- 页眉、页脚、页码字段和页码设置；
- 样式和潜在样式；
- 编号定义；
- 表格属性；
- 签名区和盖章区的段落属性；
- 脚注和尾注；
- 文档设置、兼容性设置和主题数据，除非明确有害。

## 高风险失败模式

- 从空白 DOCX 重建，看起来相似但缺少原始结构；
- 页眉页脚缺失或被扁平化；
- 分节符丢失导致分页漂移；
- 自动编号被转为普通文本；
- 签名区移动到错误页面；
- 中英文字体回退或字体设置混乱；
- 字段被转成过期的静态文字；
- 格式清理时意外改动实体内容。
- 占位符被 Word 自动拆分到多个 run，导致无法稳定替换；
- 文本变长后分页变化，但 OpenXML 结构仍然一致；
- 使用无占位符模板要求模型猜测替换位置，导致版式不可证明。

## 最低验收标准

精确套版报告必须说明：

- 源模板或母版路径；
- 输出是否基于母版生成；
- 页眉、页脚、分节、页码、样式和编号是否保留；
- 模板一致性门禁是否通过；
- 视觉渲染是否成功；
- 与母版相比的已知偏差。
