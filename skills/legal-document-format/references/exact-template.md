# 精确模板与确定性字段模式

精确模板处理意味着输出文档应继承用户提供的 DOCX 母版结构。它不是视觉仿制，也不是从空白 Word 文件重新画一个相似版式。

本文件描述的是“可证明结构一致”的确定性路径。普通用户的主路径见 `template-execution.md`。

## 模板执行优先

当用户提供模板或母版文档时：

1. 以母版 DOCX 作为格式事实来源。
2. 先理解用户的自然语言任务：生成、批量处理、套入内容、格式迁移或校验。
3. 默认按模板执行，而不是要求用户先放置字段。
4. 除非用户明确授权，否则保留模板自身的结构和元数据。
5. 交付前运行结构检查、文本检查和视觉检查。

## 确定性字段模式

如果模板已经包含明确 `{{KEY}}` 字段，或技术同事选择维护字段化模板，可以使用确定性字段模式。

该模式的当前实现是“复制模板 DOCX 包结构 + 替换文本节点”。这意味着页眉、页脚、页码字段、分节、样式、编号、页边距、段落属性和字体设置都应从模板继承，而不是重新生成。

标准流程：

1. 准备带 `{{KEY}}` 字段的模板 DOCX。
2. 用 `scripts/apply_docx_template.py` 生成候选 DOCX。
3. 用 `scripts/compare_docx_template_parity.py` 对比模板与候选，要求除文本节点内容外的 OpenXML 结构一致。
4. 用 `scripts/render_docx.sh` 渲染候选 DOCX。
5. 用 `scripts/format_gate.py --require-visual --fail-on-warning` 做发布档门禁。

示例：

```bash
./skills/legal-document-format/scripts/apply_docx_template.py \
  模板.docx 输出.docx \
  --replacements-json replacements.json

./skills/legal-document-format/scripts/compare_docx_template_parity.py \
  模板.docx 输出.docx
```

`replacements.json` 示例：

```json
{
  "TITLE": "民事起诉状",
  "CASE_NO": "(2026) 示例案号 001 号"
}
```

## 字段规则

- 字段使用 `{{KEY}}` 形式。
- 建议 `KEY` 使用大写字母、数字和下划线，例如 `{{CLAIMANT}}`。
- 默认支持替换位于同一段落内的字段；即使 Word 把一个字段拆到多个 run 或多个文本节点，也应能识别。
- 如果字段被拆到不同段落、不同表格单元格或复杂块之间，未解析字段门禁会阻断交付。
- 如果需要替换整段、表格行或复杂块，应先把模板设计为稳定字段块，再执行人工复核。

## 一致性判定

模板一致性门禁忽略正文文本节点内容差异，但会检查：

- DOCX 包内 part 集合；
- XML 结构、属性和关系；
- 页眉、页脚、页码字段和分节属性；
- 样式、编号、设置、主题和字体表；
- 非文本二进制部件的哈希。

因此，“一模一样”的工程含义是：除被明确替换的文本内容外，文档结构和版式载体不发生漂移。若替换后的文字长度导致分页变化，仍需通过渲染页检查和人工复核确认。

## 默认保留

默认保留：

- 分节和分节属性；
- 页面尺寸和页边距；
- 页眉、页脚、页码字段和页码设置；
- 样式和潜在样式；
- 编号定义；
- 表格属性；
- 签名区和盖章区的段落属性；
- 脚注和尾注；
- 文档设置、兼容性设置和主题数据，除非明确有害。

## 高风险失败模式

- 从空白 DOCX 重建，看起来相似但缺少原始结构；
- 页眉页脚缺失或被扁平化；
- 分节符丢失导致分页漂移；
- 自动编号被转为普通文本；
- 签名区移动到错误页面；
- 中英文字体回退或字体设置混乱；
- 字段被转成过期的静态文字；
- 格式清理时意外改动实体内容；
- 字段被拆分到不同段落、不同表格单元格或复杂块之间，导致无法稳定替换；
- 文本变长后分页变化，但 OpenXML 结构仍然一致；
- 对无字段模板宣称已自动完成所有精确填充，导致版式或内容定位不可证明。

## 最低验收标准

精确模板报告必须说明：

- 源模板或母版路径；
- 输出是否基于母版生成；
- 是否使用确定性字段模式；
- 页眉、页脚、分节、页码、样式和编号是否保留；
- 模板一致性门禁是否通过；
- 视觉渲染是否成功；
- 与母版相比的已知偏差。
