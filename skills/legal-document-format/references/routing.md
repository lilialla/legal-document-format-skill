# 路由

选择能满足请求的最窄路径。只有当用户要求更高保真度，或文书类型本身需要时，才升级路由。对外发布、插件分发和正式演示默认按完整发布档执行，必须具备视觉校验能力。

## L0 文本清理

适用于粘贴文本、法条、案例摘录、标题和标点清理。

应当：

- 保留法律实体内容；
- 规范明显空格和标点问题；
- 除非用户要求重组，否则保留原有顺序。

不得：

- 推断缺失的法律内容；
- 改写裁判结论、请求、金额或日期；
- 未经请求转换为 DOCX。

## L1 普通 DOCX 排版

适用于普通法律备忘录、草稿、报告或 Markdown-to-Word 任务，且不要求精确模板一致。

应当：

- 应用用户要求的字体、字号、行距、缩进、页边距和标题层级；
- 生成可读的格式报告；
- 交付质量重要时运行渲染校验。

不得：

- 把结果表述为精确模板套版；
- 凭视觉记忆仿制模板。

## L2 精确模板套版

适用于用户提供 DOCX 模板、母版裁决书、机构格式，或要求保留同款样式、页眉页脚、页面设置、分节或分页逻辑。

应当：

- 从用户提供的母版 DOCX 开始；
- 在母版结构内替换内容；
- 除非明确要求，保留样式、页眉页脚、分节、编号、页面设置和签名区。

不得：

- 创建空白 DOCX 并手工近似模板；
- 丢失分节符或页码字段；
- 扁平化页眉页脚或编号。

## L3 裁决书风格定稿

适用于仲裁裁决书、裁决式正式文书和最终定稿类文书。

应当：

- 除非用户明确授权法律修改，否则视为内容已锁定；
- 保留主文和签名区域；
- 将标题、当事人、程序经过、事实、裁判理由、裁决项、费用、日期、印章和附件作为格式面检查；
- 运行视觉校验；缺少 LibreOffice 或 Poppler 时，不得称为完整发布档。

不得：

- 格式阶段改变裁决结果；
- 添加没有依据的事实或法律理由；
- 未经律师审阅即宣称可正式提交。

## L4 视觉校验

适用于用户要求渲染、检查、比较或校验版面。

应当：

- 使用 LibreOffice 将 DOCX 渲染为 PDF；
- 使用 Poppler 将 PDF 渲染为 PNG；
- 发布档先运行 `scripts/check_release_requirements.py --mode release`；
- 发布档使用 `scripts/format_gate.py --require-visual`；
- 只有存在基准时才进行基准比较；
- 报告分页漂移、页眉页脚缺失、页码异常、编号断裂、溢出和签名区移位等风险。

不得：

- 将渲染成功等同于法律内容正确；
- 没有视觉 diff 时宣称像素级一致。
